## Diagnostics & fixes done

### Issue 1 — Hermes unreachable via Telegram after reboot  (RESOLVED, self-healed)
- Bot: `@Bazpchermes01bot` (id 8113658356) → Hermes (local Ollama: llama3.2:3b / qwen3:8b).
- Gateway runs as systemd **user** service `hermes-gateway` (pid 360, up since 08:28 boot),
  Telegram **connected, polling**. Token valid; no webhook; no phantom instance.
- **Verified outbound works**: sent live test message (message_id 20) to chat 6062064959
  (Barry The AI Dev / @BarryTheAiDev) — delivered.
- Confirmed poller is alive: a diagnostic `getUpdates` peek triggered the gateway's
  `Conflict: terminated by other getUpdates request` and it grabbed the slot back.
- **Root cause**: classic **post-hard-shutdown getUpdates conflict** — machine was powered
  off without graceful gateway shutdown, so Telegram held the old long-poll session open;
  the boot-time gateway couldn't receive until Telegram expired it. Self-heals after boot.
  (Yesterday's restarts were graceful → released the slot; a cold power-off isn't.)
- Prevention: `systemctl --user stop hermes-gateway` before powering off, or just wait a
  few min after boot.
- Note: earlier 2026-06-15 ~10:27 there were genuine `InvalidToken`/`Chat not found` errors,
  but those were fixed later that day — not today's cause.

### Issue 2 — Claude channel wouldn't start ("server error")  (FIXED)
- This is the official **Telegram plugin** (`telegram@claude-plugins-official`), separate
  bot `@Bazminipcclaude02bot` (id 8682389341). There is **no `--channels` CLI alias problem**;
  the flag is `claude --channels plugin:telegram@claude-plugins-official` (plural).
- **Root cause**: `C:\Users\barry\.claude\channels\telegram\.env` started with a **UTF-8 BOM**
  (bytes `EF BB BF`, written by PowerShell which adds a BOM by default). That made the first
  key parse as `﻿TELEGRAM_BOT_TOKEN` → token never read → MCP server threw on startup.
- **Fix applied**: stripped the BOM + normalized CRLF in place; backup at
  `~/.claude/channels/telegram/.env.bak.bom`. Token validated against Telegram
  (bot @Bazminipcclaude02bot). MCP server (`server.ts` via Bun) now boots **clean**.
- **Verified**: `claude --channels plugin:telegram@claude-plugins-official -p "…"` returned
  `CHANNEL_OK`. Bun 1.3.14 present on Windows + WSL.
- Remaining for a usable session: run with `--channels`, then **pair** (DM the bot → 6-char
  code → `/telegram:access pair <code>`), then lock down (`/telegram:access policy allowlist`).
  Access policy currently default = `pairing`, empty allowlist.
