═══ HOW A CHANNEL SESSION WORKS ═══

Each service runs a chain:
  systemd → start-*.sh → script -qfc (pseudo-TTY) → _channel-exec-*.sh
    → claude --dangerously-skip-permissions --model sonnet
         --channels plugin:telegram@claude-plugins-official
         [--append-system-prompt <soul>]

When claude starts with --channels, it spawns a BUN bridge child process:
  bun run --cwd ~/.claude/plugins/cache/claude-plugins-official/telegram/<version> start

The bun bridge is the ONLY part that polls Telegram/Discord and sends replies.
Claude's normal text output does NOT reach the user — every reply must go through
the reply MCP tool call.

KNOWN UPSTREAM BUG: The bun bridge child can die while the claude parent keeps
running. systemd sees the service as "active" so Restart=always never fires. The
node is deaf+mute but looks healthy to naive checks (systemctl is-active, ps).

═══ RELIABILITY LAYER ═══

tc-bridge-guardian.service — runs every 5s, reads /sys/fs/cgroup/.../cgroup.procs

bazment-guardian.service — similar guardian for the bazment-* family services.

═══ PLUGIN SETTINGS — CRITICAL QUIRK ═══

Each claude channel session needs:
  ~/.claude/channels/<plugin-name>/settings.json
to exist with { "enabledPlugins": {"telegram": true} } (or discord), or the bun
bridge silently fails to start. The claude process launches fine and systemd reports
"active" — but no messages are received and no replies can be sent.

This file is NOT recreated on restart by default. Running `claude plugin install`
mid-session also wipes it. Both have caused silent fleet failures.

Fix pattern: each _channel-exec-*.sh script should recreate settings.json if
missing before launching claude.
