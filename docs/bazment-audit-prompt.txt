════════════════════════════════════════════
WHAT BAZMENT IS
════════════════════════════════════════════

Bazment is a single git repo (~/bazment/) that contains everything needed to run any agent
on any host. One `git clone + install.sh` bootstraps a new host. All agents share the same
directory structure, startup scripts, guardian, and vault sync. Config differences are per-agent
files (soul.md, CLAUDE.md, MEMORY.md), not scattered scripts.

Repo: github.com/bazdev0001/bazment (private)
Location on mini-PC: ~/bazment/
Location on VPS: ~/bazment/ (partially deployed — Ada uses legacy services, not bazment services)

HOW AN AGENT STARTS (8-STEP SEQUENCE)
════════════════════════════════════════════

Step 0 — systemd fires the service
  bazment-sage-telegram.service
  → services/run-sage-telegram.sh  (sets BAZMENT_AGENT=sage, BAZMENT_ROOT, cd to channel dir)
  → services/_exec-sage-telegram.sh wrapped in: script -qfc ... /dev/null
  → loads agents/sage/soul.md via --append-system-prompt
  → exec claude --dangerously-skip-permissions --model sonnet
        --channels plugin:telegram@claude-plugins-official

Step 1 — Claude auto-loads CLAUDE.md
  agents/default/CLAUDE.md (base rules, ≤200 lines)
  agents/<name>/CLAUDE.md (agent-specific extensions)

Step 2 — Claude auto-loads MEMORY.md
  agents/default/MEMORY.md + agents/<name>/MEMORY.md

Step 3 — SessionStart hook fires agents-general-execution.sh
  Injects (by printing to stdout — Claude Code reads hook output as context):
  - obsidian/00-Briefing.md + today's journal
  - obsidian/DECISIONS.md (settled decisions — do not re-propose)
  - obsidian/notes/LEARNINGS.md + FRICTION-LOG.md (lessons + known mistakes)
  - obsidian/notes/AGENTS.md (fleet registry)
  - agents/default/rules.md (full rulebook — no 200-line cap, injected here)
  - fleet + agent MEMORY.md
  - obsidian/log/<date>.md (3h distilled conversation digest)
  - obsidian/log-raw/<date>.md (last 100 lines, raw, marked UNTRUSTED)
  - Environment map (paths, host, agent name)
  - Open GitHub issues assigned to this agent
  - Writes heartbeat to data/agents/<name>/heartbeat

Step 4 — agents/<name>/agents-execution.sh
  Injects department context, open tasks, agent-specific startup (Athena: trading signals, etc.)

THE 97% RULE (critical — agents must know this)
════════════════════════════════════════════

Before asking Barry OR another agent ANY question, complete all 8 steps:
  0. git pull — get latest vault
  1. obsidian/00-Briefing.md — in fleet state?
  2. obsidian/DECISIONS.md — already settled?
  3. Today's journal — happened today?
  4. obsidian/tasks/ — existing task covers it?
  5. obsidian/notes/LEARNINGS.md — fleet already knows?
  6. obsidian/notes/FRICTION-LOG.md — known mistake?
  7. departments/<dept>/context.md — domain answer?
  8. agents/<name>/MEMORY.md — agent-specific fact?

All 8 fail → only then ask Barry. Skipping any step → FRICTION-LOG entry.
This is called the 97% rule because Barry expects 97% of questions answered from vault without asking.

THE BUN BRIDGE BUG (critical operational knowledge)
════════════════════════════════════════════

Each claude channel session spawns a bun bridge child:
  bun run --cwd ~/.claude/plugins/cache/claude-plugins-official/telegram/<ver> start

This bun bridge is the ONLY thing that polls Telegram/Discord and sends replies.
Claude's normal text output NEVER reaches Barry — every reply must go through the reply MCP tool.

UPSTREAM BUG: The bun bridge dies mid-session while claude keeps running. systemd sees
"active" and never fires Restart=always. Node is deaf+mute but looks healthy to systemctl/ps.

PLUGIN SETTINGS FRAGILITY: The bun bridge also requires ~/.claude/channels/<plugin>/settings.json
(or bazment/channels/<agent>/<platform>/settings.json) with:
  { "enabledPlugins": {"telegram": true}, "dmPolicy": "allowlist", "allowFrom": ["6062064959"] }
Without this file the bridge silently fails to start. Running `claude plugin install` wipes it.
Each _exec-*.sh should recreate settings.json if missing before calling claude.

RELIABILITY LAYER
════════════════════════════════════════════

bazment-guardian.service — daemon, 5s loop, two detectors:
  1. DEAD BRIDGE: reads /sys/fs/cgroup/.../cgroup.procs, checks for live bun process per service.
     If missing 2 consecutive sweeps (~10s), restarts the service.
  2. FROZEN SESSION: bridge is alive but inbox has messages older than 10min = session wedged.
     Detected via inbox dir file age. Restarts the service.
  Auto-discovers which bazment-* services are enabled — no hardcoded list.

bazment-vault-sync.timer — every 5min, git pull bazment repo + pull/push obsidian vault.

bazment-ada-vps-watchdog.timer — every 5min, reads Ada's heartbeat file from shared vault,
  alerts via Telegram if heartbeat stale >30min. (NOTE: heartbeat cron NOT deployed on VPS yet.)

True health check: bash ~/bazment/scripts/fleet-status.sh
